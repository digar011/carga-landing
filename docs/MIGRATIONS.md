# Database Migration Guide

> CarGA uses Supabase (PostgreSQL) with 11 sequential migration files. All schema changes go through migrations — never modify the database manually.

## Migration Files Overview

All migrations live in `supabase/migrations/` and are applied in timestamp order.

| # | File | What It Creates |
|---|------|-----------------|
| 1 | `20250325000001_create_enums.sql` | Custom enum types: `user_role`, `load_status`, `application_status`, `truck_type`, `cargo_type`, `notification_type`, `plan_type`, `subscription_status` |
| 2 | `20250325000002_create_users_table.sql` | `users` table linking `auth.users` to app roles, RLS policies, role index |
| 3 | `20250325000003_create_profiles.sql` | `profiles_transportista` and `profiles_cargador` tables with domain-specific fields (CUIT, rating, plan, province, etc.) |
| 4 | `20250325000004_create_trucks.sql` | `trucks` table with patente validation (Argentine plate formats), capacity, and type |
| 5 | `20250325000005_create_loads.sql` | `loads` table — core load board data: origin/destination coords, cargo type, weight, rate, status lifecycle |
| 6 | `20250325000006_create_applications.sql` | `load_applications` table with unique constraint per load+transportista, status tracking |
| 7 | `20250325000007_create_ratings.sql` | `ratings` table (1-5 stars) with triggers to auto-update profile average ratings |
| 8 | `20250325000008_create_notifications.sql` | `notifications` table with type, read status, JSONB metadata, optimized indexes for unread queries |
| 9 | `20250325000009_create_subscriptions.sql` | `subscriptions` table with Mercado Pago integration fields, plan/status tracking |
| 10 | `20250325000010_create_admin_logs.sql` | `admin_logs` audit table for admin actions with entity tracking |
| 11 | `20250326000001_production_indexes_and_functions.sql` | Production optimizations: composite indexes, `pg_trgm` fuzzy search, helper functions (`count_user_loads_this_month`, `get_user_plan`, `is_user_verified`), Realtime subscriptions, data integrity triggers (`prevent_self_application`, `check_rating_eligibility`) |

## Creating a New Migration

```bash
supabase migration new description_of_change
```

This creates a new file with the format `YYYYMMDDHHMMSS_description_of_change.sql` in `supabase/migrations/`.

Example:

```bash
supabase migration new add_geofence_column
# Creates: supabase/migrations/20250327143022_add_geofence_column.sql
```

Then write your SQL in the generated file.

## Rules

1. **Never edit existing migrations.** Once a migration has been applied (locally or remotely), it is immutable. To change something, create a new migration.
2. **Always create new migrations** for schema changes, even small ones.
3. **Use `IF NOT EXISTS` / `IF EXISTS`** for indexes, extensions, and types to make migrations idempotent.
4. **Include RLS policies** for every new table — CarGA enforces RLS on all tables without exception.
5. **Add indexes** for columns used in WHERE, JOIN, or ORDER BY clauses.
6. **Test locally first** before applying to any remote environment.

## Applying Migrations

### Local Development

```bash
# Apply all pending migrations to local Supabase
make db-push
# or
supabase db push
```

### Reset Local Database (Destructive)

```bash
# Wipes everything and reapplies all migrations from scratch
make db-reset
# or
supabase db reset
```

This drops all tables and re-runs every migration in order. Use only in local development.

### Production / Staging

Apply migrations through the Supabase Dashboard or CI pipeline — never run `supabase db push` against production directly.

1. Open the Supabase Dashboard for the target project
2. Go to SQL Editor
3. Paste the migration SQL and execute
4. Verify the changes in Table Editor

Or use the CI/CD pipeline which runs `supabase db push` against the linked project automatically.

## Rollback Strategy

Supabase migrations do not support automatic rollback. To undo a migration:

1. **Create a new migration** that reverses the change:

```bash
supabase migration new rollback_geofence_column
```

2. Write the reverse SQL:

```sql
-- Rollback: remove geofence column added in 20250327143022
ALTER TABLE loads DROP COLUMN IF EXISTS geofence_radius;
DROP INDEX IF EXISTS idx_loads_geofence;
```

3. Apply the rollback migration normally.

## Naming Convention

```
YYYYMMDDHHMMSS_description.sql
```

- Timestamp is auto-generated by `supabase migration new`
- Description uses `snake_case`
- Be descriptive: `create_users_table`, `add_geofence_to_loads`, `fix_rating_trigger`

## Testing Migrations

Follow this sequence for every schema change:

1. **Local:** Apply to local Supabase, verify with `make db-types` and `make test`
2. **Staging:** Push to staging environment, run E2E tests
3. **Production:** Apply to production during low-traffic window, monitor for errors

```bash
# Full local verification workflow
make db-push           # Apply migration
make db-types          # Regenerate TypeScript types
make typecheck         # Verify types compile
make test              # Run unit tests
make e2e               # Run E2E tests
```

## Common Pitfalls

### Circular Foreign Key References

Do not create two tables that reference each other. If table A references table B and table B references table A, the migration will fail. Use a junction table or defer one of the constraints.

### Missing IF NOT EXISTS

Always use `IF NOT EXISTS` for indexes and extensions:

```sql
-- Good
CREATE INDEX IF NOT EXISTS idx_loads_status ON loads (estado);
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- Bad — will fail if the index already exists
CREATE INDEX idx_loads_status ON loads (estado);
```

### Enum Changes

PostgreSQL enums cannot have values removed. To add a new value:

```sql
ALTER TYPE load_status ADD VALUE IF NOT EXISTS 'archivada';
```

To remove or rename an enum value, you must create a new enum type, migrate the data, and drop the old one — this is complex and should be avoided when possible.

### RLS Policy Conflicts

When adding RLS policies, ensure the policy name is unique per table. Use descriptive names:

```sql
CREATE POLICY "loads_select_published"
  ON loads FOR SELECT
  USING (estado = 'publicada');
```

### Column Defaults on Existing Tables

When adding a NOT NULL column to a table with existing data, always provide a DEFAULT:

```sql
ALTER TABLE loads ADD COLUMN priority INT NOT NULL DEFAULT 0;
```
